WHAT PHASE 1.5 FILTERS APPLY TO:
Filters apply to SEARCH RESULTS — not the personal log (that 
comes in Phase 2). When she searches for bridges, she can narrow 
the results by geography, architect, or structural engineer before 
even tapping into a detail page.

FILTER SYSTEM SPEC:

1. Geographic filter
A filter option on the search results screen that lets the user 
narrow results by location. The hierarchy is:
- First level: Country (e.g. United States, France, Australia)
- Second level: If United States or Canada is selected, show a 
  State/Province picker underneath. For all other countries, 
  country level is enough.
- These filters stack with the structure type filter — she should 
  be able to filter by both country AND structure type simultaneously.
- Source for country and state: derive from the region label 
  already returned on each search result. No new API needed.

2. Filter by structure type
Already partially implemented via badges — formalize this as an 
explicit filter option on the search results screen. Select one 
or more structure types to narrow results.

3. Filter by architect
A filter option on the search results screen showing architects 
present in the current result set. Tapping one narrows the 
results to bridges by that architect.
- Only show architects present in the current search results —
  not a global list.
- Source: architect field already fetched from Wikidata/OSM.

4. Filter by structural engineer
Same as architect filter but for structural engineer. Only show 
engineers present in the current result set.
- Source: structural engineer field already fetched from 
  Wikidata/OSM.

5. Clickable architect and engineer names on the detail page
On the bridge detail page, the Architect and Structural Engineer 
fields in the facts table should be tappable. Tapping a name 
navigates back to search results with that architect or engineer 
already applied as the active filter.
- Should feel like a link, not a button — subtle underline or 
  colored text.
- If no other results exist for that person, show a clear empty 
  state: "No other bridges by [name] found."

ALSO — in Phase 2 and beyond, these same filters should carry 
over to the My Bridges personal log screen as well, so she can 
filter her saved collection the same way.

IMPLEMENTATION NOTES FOR CLAUDE CODE:
- Filter UI should be a bottom sheet or horizontal chip row — 
  mobile-first, 390px, iPhone Safari appropriate. Not a sidebar.
- All filters use AND logic — multiple filters narrow results, 
  never expand them.
- Architect and engineer filters are only shown when that data 
  is available in the result set — hide them if no results have 
  that field populated.
- Flag to user before building which UI pattern you plan to use 
  for the filter controls (bottom sheet vs chips vs dropdown) 
  and wait for approval.
- Update PRODUCT.md with this full spec, insert Phase 1.5 into 
  the phase list, and mark all filter behavior as carrying 
  forward into Phase 2+ for the personal log.

Update the country filter pinning logic:

If the user has granted location permission in their browser 
(navigator.geolocation is available and permitted), use their 
current GPS coordinates to reverse-geocode their country and 
pin that country at the top of the filter list.

If GPS is unavailable, denied, or times out — fall back to 
pinning United States at the top by default. Silent fallback, 
no error message shown to the user.

Canada always appears second when US is pinned by default. 
When GPS determines the top country, Canada does not get 
special treatment — it falls alphabetically with everything 
else unless the user is in Canada, in which case Canada is 
pinned first.

For reverse geocoding the GPS coordinates to a country name, 
use the Nominatim reverse geocoding endpoint — it's already 
in our ecosystem, free, no new API needed:
https://nominatim.openstreetmap.org/reverse?format=json&lat={lat}&lon={lon}
The country field is in the response as address.country.

Important: this is MVP web app — do not request GPS 
permission automatically on page load. Only request it when 
the user opens the filter panel and taps the Country filter. 
That way the browser permission prompt appears in context, 
not randomly when she first opens the app.
